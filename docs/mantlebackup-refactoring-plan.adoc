= mantlebackup_controller.go 段階的リファクタリング計画

== 目的

link:new-architecture.adoc[Mantle（再）設計案]に従い、`internal/controller/mantlebackup_controller.go`（約 3,500 行、以下 MB-C）を三層アーキテクチャ（ドメイン層・ユースケース層・インフラ層）にリファクタリングする。このリファクタリングにより以下を達成する：

* 本体コードの関心を分離し、Kubernetes クラスタや Ceph クラスタといった外部リソースにアクセスする部分（インフラ層）と、コントローラが次に何を行うべきかを判定する部分（ドメイン層）を分けることで、コードの見通しをよくする。ドメイン層では、どのような状況でリコンサイラがどのように振る舞うのかを明示的に書く。
* ドメイン層に対して Arrange-Act-Assert を明示した（envtest を用いない）ユニットテストを書き、リコンサイラの振る舞いを担保する。
* 実装中は https://martinfowler.com/bliki/StranglerFigApplication.html[ストラングラーフィグパターン]を活用し、レビューしやすい単位に PR を分割して進める。各 PR は単体でマージしても外部から見た振る舞いや機能を損ねず、コンパイルが通る。その代わり、PR 単体では nop になるコード（後続 PR から使われるコード）を許容する。

MantleBackupConfig コントローラで実施済みの PoC（ https://github.com/cybozu-go/mantle/pull/244[PR #244] 他。`internal/controller/domain`・`usecase`・`infra` パッケージ）のパターンを正とし、同じディレクトリ・同じ流儀で実装する。

== スコープ外

* インテグレーションテスト（envtest を用いるテスト）の追加・修正・削除。ただしシンボル移動に伴う機械的な参照先修正（import・シンボル名の変更）は許容する。
* E2E テストの追加・修正・削除。
* gRPC によって呼び出される処理（`replication.go` のサーバ側実装）の改善。ただし定数移動に伴う機械的な import 修正は許容する。
* 外部から見た振る舞いが変わるような実装。リソース名（Job/PVC/PV のプレフィックス）、ラベル・アノテーション、`.status` の遷移、メトリクスなど外部から観測可能な挙動を変えない。
* `mantlerestore_controller.go`・`garbage_collector_runner.go` など他コントローラのリファクタリング。

== 現状の構造

`MantleBackupReconciler` は 3 つのロール（standalone / primary / secondary）で動作し、`Reconcile()` がロールごとの処理に分岐する。主なサブフローと外部アクセスは次の通り：

[cols="2,3,2", options="header"]
|===
| サブフロー | 主な関数 | 外部アクセス

| 管理対象判定
| `checkManagedBackup`
| K8s（PVC・StorageClass の読み取り）

| 期限切れ処理
| `expire`, `scheduleExpire`
| K8s（MantleBackup 削除）、`expireQueueCh` による遅延 requeue

| ローカルバックアップ
| `reconcileLocalBackup`, `getSnapshotTarget`, `provisionRBDSnapshot`, `createRBDSnapshot`
| K8s（PVC/PV 読み取り、status 更新）、Ceph（`rbd snap create` ほか）

| 検証（verify）
| `verify`, `createOrUpdateVerifyJob`, `checkJobStatus`
| K8s（Job/PV/PVC 作成・状態確認）、Ceph（clone 作成）

| レプリケーション（primary）
| `replicate`, `replicateManifests`, `prepareForDataSynchronization`, `startExportAndUpload`, `startExport`, `startUpload`, `createOrUpdateExportJob`, `createOrUpdateUploadJobs`, `handleCompleted{Export,Upload}Jobs`
| gRPC（`CreateOrUpdatePVC`, `CreateMantleBackup`, `ListMantleBackup`, `SetSynchronizing`）、K8s（Job/PVC 作成）、オブジェクトストレージ（`Exists`, `GetSize`）

| インポート（secondary）
| `startImport`, `reconcileZeroOutJob`, `reconcileImportJob`, `createStaticPV(C)IfNotExists`, `lockVolume`/`unlockVolume`, `markSecondarySnapshotCaptured`
| K8s（Job/PV/PVC 作成）、Ceph（`rbd lock add/ls/rm`）、オブジェクトストレージ（`Exists`）

| クリーンアップ・ファイナライズ
| `primaryCleanup`, `secondaryCleanup`, `finalizeStandalone`, `finalizeSecondary`, `removeRBDSnapshot`, `deleteMiddleSnapshots`
| K8s（Job/PVC/PV 削除）、Ceph（`rbd snap rm` ほか）、オブジェクトストレージ（`Delete`）
|===

補足事項：

* 全サブフローは `internal/controller/internal/reconcile` パッケージの `*reconcile.Result`（succeeded / requeue / error の三値）を返す。
* MB-C の先頭で定義されているラベル・アノテーション・sync-mode などの定数は、スコープ外の `replication.go` からも参照されている（primary↔secondary 間のワイヤ契約）。
* `Make*Name`・`ExtractPartNumFrom*`・`searchForDiffOriginMantleBackup`・`calculateExportDataPVCSize` などの純粋関数が多数存在し、これらは MB-C とテスト以外から参照されていない。
* `exportedDiffSizeCache`（メトリクス用のメモリ内キャッシュ）と `expireQueueCh`（遅延 requeue 用チャネル）という 2 つの内部状態を持つ。

== 目標構造

リファクタリング完了後の構成は次の通り。

ドメイン層（`internal/controller/domain`）::
ロールごとのドメインリコンサイラを定義する：
+
* `MBStandaloneReconciler`
* `MBPrimaryReconciler`
* `MBSecondaryReconciler`
+
それぞれが `Provision`・`Finalize` メソッドを持ち、new-architecture.adoc の指示通り「`if 条件 { return 処理 }` の逐次実行」スタイルで記述する。standalone と primary が共有するローカルバックアップ処理は、ドメイン層内の共通関数（または両者に埋め込む共通構造体）として括り出す。リコンサイル対象の `MantleBackup` への変更はメモリ上で行い、それ以外の外部作用はドメインイベント（`domain.Operation`）として `ReconcilerOperations` に蓄積するか、ドメイン層で定義した narrow interface を介して行う（使い分けは <<d3>> 参照）。

ユースケース層（`internal/controller/usecase`）::
ロールごとに `ReconcileMBStandalone`・`ReconcileMBPrimary`・`ReconcileMBSecondary` を定義し、それぞれの `Run` が「必要リソースの取得 → `DeletionTimestamp` による `Provision`/`Finalize` の呼び分け → 変更された `MantleBackup` の永続化 → Operations のインフラ層への引き渡し」を行う。PoC の `ReconcileMBCPrimary.Run` と同じ流儀。

インフラ層（`internal/controller/infra` ほか）::
`infra.KubernetesClient.ApplyReconcilerOperations` に MantleBackup 用の Operation の実行を追加する。Ceph 操作は既存の `ceph.CephCmd`（インターフェース＋gomock 生成済み）を、オブジェクトストレージ操作は既存の `objectstorage.Bucket`（インターフェース＋モックあり）を、gRPC 操作は `proto.MantleServiceClient` をそれぞれ実装として利用する。`mantlebackup_controller.go` は最終的に `SetupWithManager` と、ロールに応じてユースケース層の `Run` を呼ぶだけの薄い `Reconcile` のみになる。

== 主要な設計判断

[#d1]
=== D1: リコンサイル結果（requeue）の表現

ドメイン層の `Provision`・`Finalize` およびその下請け関数は、既存の `*reconcile.Result` をそのまま戻り値として使う。

* MBC PoC のドメイン層は plain `error` を返すが、MB では「成功として終了」「requeue」「エラー」の三値の表現が必須（`expire` の要求経路、`checkPVCBound` の pending 待ちなど）。
* `internal/controller/internal/reconcile` は他の層に依存しない小さなユーティリティであり、ドメイン層から依存しても new-architecture.adoc の「ドメイン層は他のどの層にもコード上の依存がない」という制約（層への依存の禁止）には反しない。PoC のドメイン層も controller-runtime（`controllerutil`）に依存している前例がある。
* 既存サブフローの戻り値型と一致するため、移行時の書き換え量を最小化できる。

ユースケース層の `Run` は `*reconcile.Result` を返し、インフラ層（`Reconcile`）で `ToCtrlResult()` により `ctrl.Result` へ変換する。

[#d2]
=== D2: 共有定数・純粋ヘルパの移動

MB-C 先頭のラベル・アノテーション・sync-mode・名前プレフィックス定数、および `Make*Name`・`ExtractPartNumFrom*` などの純粋ヘルパ関数は、ドメイン層（`internal/controller/domain`）に移動する。

* これらは「Mantle がどのように振る舞うか」の定義そのものであり、ドメイン層の管轄。
* `replication.go`・既存 envtest テストからの参照は import 変更のみの機械的修正で追従する（PR 1 で実施）。
* `getCephClusterIDFromPVC`（`util.go`）は `mantlerestore_controller.go`・`replication.go` とも共有しているため移動しない。

[#d3]
=== D3: Operations とインターフェースの使い分け

new-architecture.adoc の方針通り、**書き込み系はドメインイベント（Operation）を優先**し、**呼び出し結果がその場のリコンサイル判定に影響するものだけ**ドメイン層定義の narrow interface を使う。読み取り系は原則ユースケース層が事前に取得してドメイン層へデータとして渡す。分類は次の通り：

[cols="3,2,3", options="header"]
|===
| 外部アクセス | 方式 | 備考

| リコンサイル対象 `MantleBackup` の変更（ラベル・finalizer・アノテーション・status・conditions）
| メモリ上で変更し、ユースケース層が永続化
| PoC と同じ。ただし <<risk-persist-order>> の例外あり

| Job・PVC・PV の作成／削除、他の `MantleBackup` の更新（diff-from アノテーションなど）、期限切れ backup の削除
| Operation
| Job マニフェスト構築はインフラ層に置く（<<d4>>）

| gRPC `CreateOrUpdatePVC` → `CreateMantleBackup`（レプリケーション）
| 複合 Operation
| 前者のレスポンス（PVC UID）を後者のリクエストが使うが、リコンサイル判定には影響しないため、2 呼び出しをまとめた 1 つの Operation としてインフラ層で実行する

| gRPC `SetSynchronizing`
| Operation
| 同一パス内の後続処理はレスポンスに依存しない

| gRPC `ListMantleBackup`（セカンダリの backup 一覧）
| ユースケース層で事前取得
| `IsSynced()` でない場合のみ取得する（無条件取得にすると同期済み backup のリコンサイルごとに gRPC 呼び出しが増えるため、条件はユースケース層で判定する）

| Job の完了状態・ラベル付き Job/PVC の一覧
| ユースケース層で事前取得
| `checkJobStatus`・`handleCompletedJobsOfComponent`・`canNewJobBeCreated` 相当の判定はドメイン層で行い、入力（Job のリスト・状態）はデータとして渡す

| オブジェクトストレージ `Exists`（アップロード確認）・`GetSize`（メトリクス）
| ユースケース層で事前取得
| 必要な状態のときだけ取得するよう、取得条件をユースケース層で判定する

| RBD スナップショットの作成（作成結果の ID・サイズ・タイムスタンプを status に書く）
| ドメイン層定義の interface
| 呼び出し結果がリコンサイル内容を決めるため。`CreateAndFindRBDSnapshot` のような専用 interface をドメイン層に定義し、インフラ層が `ceph.CephCmd` で実装する

| RBD スナップショットの削除（存在確認 → 削除 → 削除確認、`removeRBDSnapshot`）
| ドメイン層定義の interface
| 読み取り結果で分岐する一連の処理のため

| RBD ボリュームロック（`lockVolume`/`unlockVolume`）
| ドメイン層定義の interface
| 同一リコンサイルパス内でロックを保持したまま後続処理を行う必要があり、Operation では表現できない

| 中間スナップショット削除・verify 用イメージ削除（Ceph）
| ドメイン層定義の interface
| 一覧取得 → 判定 → 削除の複合処理のため
|===

interface は new-architecture.adoc の指示通り「その処理に必要なメソッドのみ」を定義し、他の処理と共有しない。テストでは gomock（`ceph_mock.go` 相当）ではなく、必要に応じてテスト内で小さな fake を書く。

[#d4]
=== D4: Job マニフェスト構築の置き場所

`createOrUpdateExportJob`（約 180 行）・`createOrUpdateUploadJobs`・`createOrUpdateImportJob`（約 215 行）・`createOrUpdateZeroOutJob`・`createOrUpdateVerifyJob` の Job spec 構築コードは**インフラ層**に置く。

* ドメイン層が発行する Operation（例：`CreateOrUpdateExportJobOperation`）には、判定に由来するパラメータ（Job 名、対象 backup の情報、pool/image/snapshot 名、part 番号、full/incremental、転送サイズなど）のみを持たせる。
* Pod spec の詳細（コンテナ定義、env、securityContext、スクリプト、proxy 設定、`podImage`・`envSecret`）は外部システム（Kubernetes）への依存の詳細であり、インフラ層の Operation 実行器が組み立てる。PoC の `CreateOrUpdateMBCCronJobOperation` → `createOrUpdateMBCCronJob` と同じ関係。
* これによりドメイン層のユニットテストは「どの Job をどのパラメータで作ろうとしたか」の検証に集中でき、Job spec の細部はインテグレーション・E2E テスト（既存）に委ねる。

[#d5]
=== D5: その他の配置

`exportedDiffSizeCache`::
メトリクス算出用のメモリ内キャッシュであり、リコンサイルの判定には使われない。primary のユースケース層（`ReconcileMBPrimary` のフィールド）に置く。

`expireQueueCh`・`scheduleExpire`::
workqueue への遅延投入はインフラ層の関心。ドメイン層は「まだ期限切れでない」ことを判定して `ScheduleExpireOperation` を発行し、インフラ層の実行器がチャネルへ送出する。`SetupWithManager` の `WatchesRawSource` 配線は現状のまま `mantlebackup_controller.go` に残す。

`prepareObjectStorageClient`（S3 クライアントの遅延初期化）::
インフラ層に移す。ConfigMap/Secret から認証情報を読んで `objectstorage.Bucket` を返す処理をインフラ層のコンポーネントとし、ユースケース層はそれを介して `Exists`/`GetSize` の事前取得を行う。

メトリクス出力::
PoC（`MBCPrimaryReconciler.Provision`）と同様、ドメイン層から `metrics` パッケージを直接呼んでよい（外部システムへの依存がないため）。new-architecture.adoc も「メトリクスの出力など、この制約に馴染まない処理は存在しうる」と明記している。

ユースケース層の初期化::
MBC PoC と同様、`MantleBackupReconciler.Reconcile` 内で `sync.Once` により遅延初期化する（既存 envtest スイートの制約。PoC の TODO コメント参照）。

== 段階的移行計画

=== 移行戦略

ストラングラーフィグパターンに基づき、次の方針で進める：

1. **既存のオーケストレーション（`reconcileAsStandalone` などの旧メソッド群）をファサードとして残したまま**、サブフローを 1 つずつドメイン層の関数（＋ユニットテスト）に置き換える。旧メソッドは「入力の取得 → ドメイン関数の呼び出し → Operations の即時適用」をするだけの形に痩せていく。
2. あるロールの全サブフローがドメイン化された時点で、そのロールの `Provision`/`Finalize`（逐次 `if` の形）とユースケース層の `Run` を組み立て、`Reconcile` の該当ロール分岐を切り替え、不要になった旧コードを削除する。
3. ロールは standalone → primary → secondary の順に完成させる。standalone のフローは primary の部分集合（ローカルバックアップ＋verify＋cleanup）なので、共有部分のドメイン化がそのまま primary の下準備になる。

この戦略により、切り替え PR 以外のどの時点でも旧経路が現役で動き続け、各 PR のレビュー範囲が「1 サブフローのドメイン化とそのテスト」に限定される。

=== PR 一覧

依存関係は原則「番号順」だが、PR 8〜10（primary 系）と PR 11〜12（secondary 系）は互いに独立しており並行して進められる。

==== PR 1: 純粋ヘルパ・共有定数のドメイン層への移動

* **内容**: MB-C 先頭の定数群（ラベル・アノテーション・sync-mode・名前プレフィックス）と、外部アクセスを持たない純粋関数（`Make*Name`、`ExtractPartNumFrom*`、`IsPartNextToLargestCompletedPart`、`isCreatedWhenMantleControllerWasSecondary`、`convertToMap`、`searchForDiffOriginMantleBackup`、`calculateExportDataPVCSize`、`isPVSmallerThanPVC` など）を `internal/controller/domain` に移動する。`replication.go`・既存テストの参照は機械的に修正する。
* **nop**: なし（移動のみ）。
* **テスト**: 移動した純粋関数のうちテストが薄いものに AAA ユニットテストを追加する（特に `searchForDiffOriginMantleBackup`、`ExtractPartNumFrom*`）。既存の関連テストが `internal/controller` 側にあればテスト関数ごと domain 側へ移す。
* **規模**: 大きめだが、ほぼ機械的な移動 diff。

==== PR 2: MantleBackup 用 Operation 基盤

* **内容**: MantleBackup のリコンサイルで使う Operation 型の骨格（第 1 弾として PR 3〜6 が使う分：`DeleteMantleBackupOperation`、`ScheduleExpireOperation`、`CreateOrUpdateVerifyJobOperation`、静的 PV/PVC 作成、verify リソース削除系）と、`infra.ApplyReconcilerOperations` への実行器の追加を行う。実行器がチャネル（`expireQueueCh`）や `ceph.CephCmd` を必要とするため、`infra.KubernetesClient` の依存注入を拡張する。
* **nop**: この PR の時点では Operation を発行するコードが存在しないため、追加分は全て nop。
* **テスト**: Operation 型は構造体のみのためユニットテスト対象外（実行器はインテグレーション・E2E の守備範囲）。
* **規模**: 小。
* **備考**: 以降の PR で新しい Operation 型が必要になった場合は、その Operation を初めて発行する PR で型と実行器を一緒に追加する（巨大な nop PR を避ける）。

==== PR 3: expire と checkManagedBackup のドメイン化

* **内容**: `expire` の判定ロジック（CreatedAt 未設定・deletionTimestamp・retain-if-expired アノテーション・期限計算）をドメイン層の関数に移し、削除は `DeleteMantleBackupOperation`、遅延 requeue は `ScheduleExpireOperation` として発行する。`checkManagedBackup` の判定部分（ラベル一致・cluster ID 照合）もドメイン化し、PVC と cluster ID の取得は呼び出し側に残す。旧 `expire`・`checkManagedBackup` はドメイン関数を呼んで Operations を即時適用する形に書き換える。
* **テスト**: 「期限切れなら削除 Operation を発行する」「retain-if-expired なら何もしない」「未期限なら schedule Operation を発行する」等の AAA ユニットテスト。
* **規模**: 小〜中。最初のサブフロー移行としてパターンを確立する PR。

==== PR 4: ローカルバックアップ準備とスナップショット作成のドメイン化

* **内容**:
** `reconcileLocalBackup` のうち、ラベル（cluster-id / local-backup-target-pvc-uid）と finalizer の付与、`getSnapshotTarget` の判定部分（PVC UID 照合、`checkPVCBound`、PV からの image/pool 抽出）をドメイン化する。PVC・PV の取得は呼び出し側に残す。
** `provisionRBDSnapshot` を「2 周のリコンサイル」に分解してドメイン化する（<<risk-persist-order>> 参照）：第 1 周は PVC/PV マニフェストを status にセットして requeue（永続化はユースケース層／旧呼び出し側）、第 2 周はマニフェスト永続化済みを確認してから `CreateAndFindRBDSnapshot` interface（この PR でドメイン層に定義、インフラ層で `ceph.CephCmd` により実装）でスナップショットを作成し、SnapID・CreatedAt・SnapSize・conditions をメモリ上で更新する。
* **テスト**: ラベル・finalizer 付与、マニフェスト未保存時の requeue、スナップショット作成後の status 更新、fake interface を使った作成失敗時の挙動。
* **規模**: 中。ドメイン interface を使う最初の PR。

==== PR 5: verify のドメイン化

* **内容**: `verify` を「clone 作成（interface）→ 静的 PV/PVC 作成（Operation）→ verify Job 作成（Operation）→ Job 完了状態（入力データ）に応じた Verified condition の更新（メモリ上）」の逐次 `if` に書き換える。verify Job の完了状態は呼び出し側が `checkJobStatus` 相当で取得してデータとして渡す。`createOrUpdateVerifyJob` の Job spec 構築と `createStaticPV(C)IfNotExists` の本体はインフラ層の Operation 実行器へ移す。
* **テスト**: Job 未完了なら何もしない、成功／失敗に応じた condition 設定、各 Operation の発行内容。
* **規模**: 中。

==== PR 6: primaryCleanup と finalizeStandalone のドメイン化

=== 各 PR 共通の完了条件（Definition of Done）

* `make build`・`make test`（lint 含む）が通る。
* 既存のインテグレーションテスト（envtest）・E2E テストに実質的変更がない（機械的な参照先修正を除く）。
* 追加したドメイン層コードに AAA ユニットテストがあり、`// Arrange`・`// Act`・`// Assert` コメントを明示している。テストは plain `go test` + testify、`t.Parallel()`、functional options によるフィクスチャビルダ（`domain/mbc_primary_reconciler_test.go` のスタイル）に従う。
* nop コードを含む場合、どの後続 PR で使用予定かを PR 説明に明記する。
* 外部から観測可能な挙動（リソース名・ラベル・アノテーション・status 遷移・メトリクス）に変化がない。

== リスク・注意点

[#risk-persist-order]
=== status 永続化と Ceph 操作の順序不変条件

`provisionRBDSnapshot` は「PVC/PV マニフェストを status に**永続化した後に** RBD スナップショットを作成する」ことでクラッシュ時の孤児スナップショットを防いでいる（MB-C 内コメント参照）。「変更をメモリ上に溜めて `Run` の最後に一括永続化」する標準パターンではこの不変条件が壊れる。PR 4 の通り、リコンサイルを 2 周（マニフェスト永続化 → 次周でスナップショット作成）に分けることで、不変条件を保ちながらドメイン化する。リコンサイルが 1 周増えるが、外部から見た状態遷移の順序は変わらない。同様に「途中で永続化してから次の外部操作を行う」箇所が他にないか、各 PR で移行対象コードを精査すること。

=== 同一パス内で結果に依存する外部呼び出し

`lockVolume` → Job 処理 → `unlockVolume`（ロック保持）、`createRBDSnapshot`（作成結果を status へ反映）などは Operation では表現できない。これらは無理に Operation 化せず、ドメイン層定義の narrow interface で扱う（<<d3>>）。Operation 化と interface 化の判断に迷ったら「呼び出し結果がその場の判定・出力に使われるか」を基準にする。

=== 事前取得によるアクセス増加

読み取りをユースケース層の事前取得に変えると、従来は条件付きだった K8s/gRPC/S3 アクセスが毎回発生しうる。K8s の読み取りはキャッシュ経由のため増加は許容するが、**gRPC・S3・`rbd` コマンドの呼び出し回数は増やさない**こと。ユースケース層で従来と同じ条件（例：`IsSynced()`、conditions の状態）を確認してから取得する。

=== 切り替え PR（PR 7・10・13）のリスク

ロール切り替えは経路の一括変更であり、サブフロー移行 PR よりリスクが高い。切り替え PR ではロジックの変更を一切含めず「組み立て・配線・削除」に限定し、既存のインテグレーション・E2E テストの通過をもって等価性を確認する。デグレを検知しやすくするため、切り替え PR の前にそのロールのドメイン層ユニットテストが分岐を網羅していることを確認する。

=== requeue タイミングの等価性

旧コードの `reconcile.Requeue()` は `REQUEUE_RECONCILIATION_AFTER` 環境変数由来の遅延を持つ。ドメイン化の際に requeue の発生箇所・回数が変わると、バックアップ完了までの時間が観測可能に変わりうる。原則として旧コードと同じ箇所で同じ種別の結果を返すこと。

=== 定数移動のワイヤ契約

PR 1 で移動する定数は primary↔secondary 間の通信・リソース契約であり、**値の変更は一切禁止**。移動時は文字列リテラルの diff がないことをレビューで確認する。
